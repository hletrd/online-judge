## F1 — audit-logs and login-logs use `Number()` for query params instead of `parseInt`
- **Severity:** MEDIUM | **Confidence:** HIGH
- **File:** `src/app/api/v1/admin/audit-logs/route.ts:47-48`, `src/app/api/v1/admin/login-logs/route.ts:34-35`
- `Number(searchParams.get("page") ?? "1")` produces `NaN` when `page` is a non-numeric string like `abc`, then `Math.floor(NaN)` is `NaN`, and `Math.max(1, NaN)` is `NaN` (since `Math.max` with any `NaN` returns `NaN`). This means a non-numeric `page` query param results in an invalid offset computation.
- Same bug class as the previously fixed anti-cheat NaN issue (commit 88391c26). The tags route was already fixed to use `parseInt` with `||` fallback in cycle 1, but these two admin routes were missed.
- **Fix:** Use `parseInt` with fallback: `parseInt(searchParams.get("page") ?? "1", 10) || 1`

## F3 — Practice page fetches all problem IDs for progress filter (Path B), potential memory issue
- **Severity:** MEDIUM | **Confidence:** MEDIUM
- **File:** `src/app/(public)/practice/page.tsx:410-447`
- When a progress filter is active and the user is logged in, Path B fetches ALL problem IDs (`allProblemRows`) and ALL user submissions for those problems, then filters in-memory. For a large problem set (thousands of problems), this loads all IDs into memory every request.
- **Fix:** Move the progress filtering into SQL using a CTE or subquery rather than fetching all IDs and filtering in JS.

- `recordRateLimitFailure` and `recordRateLimitFailureMulti` appear to be unused — all call sites now use `consumeRateLimitAttemptMulti` which does atomic check+record.

## F5 — Rankings page uses raw SQL with string interpolation for period clause
- **Severity:** LOW | **Confidence:** HIGH
- **File:** `src/app/(public)/rankings/page.tsx:31-39`
- `getPeriodClause` returns a raw SQL fragment via string concatenation into the query. The current implementation is safe because the period values are hardcoded in `PERIOD_FILTER_VALUES` and validated before use, but the pattern is fragile — a future developer could accidentally pass unsanitized input.
- **Fix:** Consider using Drizzle's query builder for the period filter instead of raw SQL string interpolation.

## F6 — `recordRateLimitFailure` uses `nanoid()` for id but `consumeRateLimitAttemptMulti` does not
- **Severity:** LOW | **Confidence:** MEDIUM
- **File:** `src/lib/security/rate-limit.ts:168,210`
- `recordRateLimitFailure` generates `nanoid()` for the `id` field on insert, but `consumeRateLimitAttemptMulti` does not include an `id` in its insert. If the `rateLimits` table requires an `id` column, the `consumeRateLimitAttemptMulti` insert could fail. However, this may be auto-generated by the DB — needs verification.
- **Fix:** Either both should include `id` or neither should (rely on DB default).
